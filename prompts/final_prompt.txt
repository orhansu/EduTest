2. Satır içi formüller için DİREKT olarak `\(` ve `\)` sembolleri kullanılmalıdır (Örn: `\( x^2 + y^2 = z^2 \)`). Kesinlikle `$` sembolü KULLANILMAMALIDIR.
3. Blok formüller için DİREKT olarak `\[` ve `\]` kullanılmalıdır (Örn: `\[ \frac{-b \pm \sqrt{b^2 - 4ac}}{2a} \]`). Kesinlikle `$$` sembolü KULLANILMAMALIDIR.

    Bu yüzden üreteceğiniz JSON dosyasında formüllerin görünümü şu şekilde OLMALIDIR:
    - Doğru Hali: "Entropi \\( \\log_a(x) \\) formülüyle...", "Karekök \\( \\sqrt{x} \\)"
    - Yanlış Hali: "Entropi \\\\( \\\\log_a(x) \\\\) formülüyle...", "Karekök \\\\( \\\\sqrt{x} \\\\)"

    - Karekök ve N. Dereceden Kök: `\\( \\sqrt{x^2+1} \\)` veya `\\( \\sqrt[3]{x-2} \\)`

    - Türev/İntegral: `\\( \\frac{dy}{dx} \\)`, `\\( \\int f(x)dx \\)`, `\\( \\sum_{n=1}^{\\infty} \\frac{1}{n^2} \\)`
    - Diğerleri: `\\( x \\geq 0 \\)`, `\\( x \\neq y \\)`, `\\( A \\cup B \\)`
    - Eşitsizlikler: Büyüktür/Küçüktür `\\( x < y \\)`, Büyük Eşit `\\( x \\geq 0 \\)`, Küçük Eşit `\\( y \\leq 10 \\)`, Eşit Değil `\\( x \\neq y \\)`

## Kapsam

3. Kod bloğu JSON string içinde olduğu için tırnak işaretleri `\"` şeklinde kaçırılmalıdır.

## Soru Konuları (Öncelik Sırası)
- Veri tipleri, değişken tanımlama ve tür dönüşümleri (casting)
- Kontrol yapıları: `if/else`, `switch/case`, `for`, `foreach`, `while`, `do-while`
- Metotlar: parametre, geri dönüş tipi, `ref`/`out`/`params` anahtar kelimeleri
- OOP: class, struct, interface, abstract class, inheritance, polymorphism, encapsulation
- Erişim belirleyiciler: `public`, `private`, `protected`, `internal`, `protected internal`
- Özellikler (Properties): auto-property, get/set, init
- İstisna yönetimi: `try/catch/finally`, özel exception sınıfları
- LINQ (Language Integrated Query):
  - **Method Syntax (Fluent):** `.Where()`, `.Select()`, `.OrderBy()` / `.OrderByDescending()`, `.GroupBy()`, `.Join()`, `.SelectMany()`, `.Distinct()`, `.Take()` / `.Skip()`, `.First()` / `.FirstOrDefault()`, `.Single()` / `.SingleOrDefault()`, `.Any()` / `.All()`, `.Count()`, `.Sum()`, `.Min()` / `.Max()` / `.Average()`, `.ToList()` / `.ToArray()` / `.ToDictionary()`
  - **Query Syntax (SQL benzeri):** `from … in … where … orderby … select …`, `group … by … into …`, `join … on … equals …`
  - **Deferred (Ertelenmiş) vs Immediate (Anlık) Execution:** `IEnumerable<T>` dönen sorgular (`Where`, `Select` vb.) listeye dönüştürülene veya enumerate edilene kadar çalışmaz (deferred); `.ToList()`, `.Count()`, `.Sum()` gibi operatörler sorguyu hemen çalıştırır (immediate).
  - **IEnumerable\<T\> vs IQueryable\<T\>:** `IEnumerable` bellekte (LINQ to Objects), `IQueryable` ise veri kaynağına özel sorgular üreterek çalışır (ör. LINQ to SQL / Entity Framework — sorgu SQL'e dönüştürülür).
  - **Lambda ve Delegate:** Lambda ifadeleri (`x => x > 5`) LINQ sorgularının omurgasıdır; `Func<T, bool>` ve `Expression<Func<T, bool>>` farkını test eden sorular üretilebilir.
- Generic tipler: `List<T>`, `Dictionary<TKey, TValue>`, `IEnumerable<T>`
- Asenkron programlama: `async/await`, `Task`, `Task<T>`
- Nullable tipler: `int?`, `string?`, null-coalescing `??`, null-conditional `?.`
- Delegates, Events, Lambda ifadeleri
- Namespace, using direktifleri, assembly kavramı

# CSS (Cascading Style Sheets) Kuralları

## Kapsam
Bu modül, CSS3 ve modern web stillendirme teknolojilerine ait test sorularının nasıl üretileceğini tanımlar.

## Kod Gösterimi
1. CSS kod parçacıkları JSON içinde `"code"` alanına string olarak yerleştirilmeli, satır sonları `\n` ile ifade edilmelidir.
2. CSS property değerlerindeki tırnak işaretleri `\"` olarak kaçırılmalıdır.
3. HTML + CSS birlikte gösterilecekse iki ayrı blok olarak `"html"` ve `"css"` alanlarında sunulabilir.
4. Girintileme 2 boşluk ile yapılmalıdır.

## Soru Konuları (Öncelik Sırası)
- Seçiciler (Selectors): element, class (`.`), id (`#`), attribute (`[attr]`), pseudo-class (`:hover`, `:nth-child`), pseudo-element (`::before`, `::after`), birleşik seçiciler (descendant, child `>`, sibling `~`, adjacent `+`)
- Özgüllük (Specificity) ve Basamak (Cascade): `!important`, hesaplama kuralları (0,1,0 notasyonu)
- Kutu Modeli (Box Model): `content`, `padding`, `border`, `margin`, `box-sizing: border-box` vs `content-box`
- Display: `block`, `inline`, `inline-block`, `none`, `flex`, `grid`, `contents`
- Flexbox: `flex-direction`, `justify-content`, `align-items`, `flex-wrap`, `flex-grow/shrink/basis`, `order`
- CSS Grid: `grid-template-columns/rows`, `grid-area`, `gap`, `fr` birimi, `auto-fill/auto-fit`
- Konumlandırma (Positioning): `static`, `relative`, `absolute`, `fixed`, `sticky`, `z-index`
- Responsive Tasarım: `@media` sorguları, `min-width/max-width`, `vw/vh` birimleri, `clamp()`
- Tipografi: `font-family`, `font-weight`, `line-height`, `letter-spacing`, `text-transform`, web fontları
- CSS Değişkenleri (Custom Properties): `--variable-name`, `var()` kullanımı
- Animasyon ve Geçiş: `transition`, `animation`, `@keyframes`, `transform`
- Renk ve Degradeler: `rgb()`, `hsl()`, `opacity`, `linear-gradient`, `radial-gradient`
- Modern CSS: `:is()`, `:where()`, `:has()`, `aspect-ratio`, `gap` (flex/grid), container queries

## Kod Örneği JSON Formatı
```json
{
  "id": 1,
  "text": "Aşağıdaki CSS kuralı uygulandığında .kutu elementinin toplam genişliği kaç piksel olur?",
  "code": ".kutu {\n  width: 200px;\n  padding: 10px;\n  border: 5px solid black;\n  box-sizing: content-box;\n}",
  "choices": [
    { "label": "A", "text": "200px", "correct": false },
    { "label": "B", "text": "210px", "correct": false },
    { "label": "C", "text": "220px", "correct": false },
    { "label": "D", "text": "230px", "correct": true },
    { "label": "E", "text": "240px", "correct": false }
  ],
  "explanation": "content-box modelinde toplam genişlik = width + sol padding + sağ padding + sol border + sağ border = 200 + 10 + 10 + 5 + 5 = 230px."
}
```

## Önemli Kurallar
- Specificity hesaplama soruları için (0,1,0) gibi açık notasyon kullanılmalıdır.
- `box-sizing` değeri belirtilmemiş sorularda varsayılan `content-box` olduğu hatırlatılmalıdır.
- Flexbox ve Grid sorularında parent-child ilişkisi kodda açıkça gösterilmelidir.
- "Hangi tarayıcıda çalışmaz?" tarzı sorular yerine standart CSS davranışı üzerine yoğunlaşılmalıdır.

# UML (Unified Modeling Language) Kuralları

## Kapsam
Bu modül, UML diyagramlarına ve nesne yönelimli tasarım kavramlarına ait test sorularının nasıl üretileceğini tanımlar.

## Diyagram Gösterimi
1. UML diyagramları görsel çizilebiliyorsa TikZ ile (`tikz` alanına) yerleştirilmelidir; aksi hâlde metin tabanlı ASCII gösterim veya PlantUML benzeri sözdizimi `"code"` alanında kullanılabilir.
2. Sınıf diyagramlarında bölümler `+`, `-`, `#`, `~` erişim belirleyicileriyle gösterilmelidir.
3. PlantUML sözdizimi tercih edildiğinde `@startuml` / `@enduml` bloğu `\n` ile satırlara bölünür.

## Soru Konuları (Öncelik Sırası)

### Yapısal Diyagramlar
- **Sınıf Diyagramı (Class Diagram):** Sınıf, interface, abstract class; ilişkiler: kalıtım (inheritance `--|>`), gerçekleme (realization `..|>`), bağlantı (association `-->`), bağımlılık (`..>`), bütünleşme (aggregation `o--`), bileşim (composition `*--`); çokluk (multiplicity: `1`, `0..*`, `1..*`)
- **Nesne Diyagramı (Object Diagram):** Belirli bir andaki nesne örnekleri ve bağlantıları
- **Bileşen Diyagramı (Component Diagram):** Sistem bileşenleri ve arayüzler
- **Dağıtım Diyagramı (Deployment Diagram):** Donanım/yazılım yerleşimi

### Davranışsal Diyagramlar
- **Kullanım Durumu Diyagramı (Use Case Diagram):** Aktör, use case, `<<include>>`, `<<extend>>`, sistem sınırı
- **Sıralı Diyagram (Sequence Diagram):** Nesne yaşam çizgisi, mesaj (senkron `->`, asenkron `->>`), dönüş (`-->`), `alt`, `loop`, `ref` çerçeveleri
- **Durum Diyagramı (State Diagram):** Durum, geçiş, olay/eylem, başlangıç/bitiş durumu, bileşik durum
- **Etkinlik Diyagramı (Activity Diagram):** Eylem, karar (diamond), çatal/birleşim, başlangıç/bitiş nodları, yüzme şeritleri (swimlane)
- **İletişim Diyagramı (Communication Diagram):** Nesneler arası mesaj sıralaması

### Tasarım Desenleri (Design Patterns)
- **Yaratımsal:** Singleton, Factory Method, Abstract Factory, Builder, Prototype
- **Yapısal:** Adapter, Decorator, Facade, Composite, Proxy, Bridge, Flyweight
- **Davranışsal:** Observer, Strategy, Command, Iterator, Template Method, State, Chain of Responsibility

## Kod Örneği JSON Formatı (PlantUML tabanlı)
```json
{
  "id": 1,
  "text": "Aşağıdaki sınıf diyagramında Kedi ve Kopek sınıfları arasındaki ilişki türü nedir?",
  "code": "@startuml\nclass Hayvan {\n  +ad: String\n  +sesCikar(): void\n}\nclass Kedi {\n  +miyavla(): void\n}\nclass Kopek {\n  +havla(): void\n}\nHayvan <|-- Kedi\nHayvan <|-- Kopek\n@enduml",
  "choices": [
    { "label": "A", "text": "Bağımlılık (Dependency)", "correct": false },
    { "label": "B", "text": "Bileşim (Composition)", "correct": false },
    { "label": "C", "text": "Kalıtım (Inheritance)", "correct": true },
    { "label": "D", "text": "Gerçekleme (Realization)", "correct": false },
    { "label": "E", "text": "Bütünleşme (Aggregation)", "correct": false }
  ],
  "explanation": "Kedi ve Kopek, Hayvan sınıfından kalıtım (generalization / inheritance) yoluyla türemektedir. UML'de bu ilişki içi boş üçgen ok (<|--) ile gösterilir."
}
```

## Önemli Kurallar
- Kalıtım ile gerçekleme (interface implementasyonu) ayrımını sorgulayan sorular etkili tuzak sorulardır.
- Bütünleşme (aggregation) ile bileşim (composition) arasındaki "sahiplik / yaşam döngüsü" farkı açıklamalarda vurgulanmalıdır.
- Tasarım deseni sorularında hangi problemi çözdüğü ve hangi prensibe dayandığı (SOLID) açıklamada belirtilmelidir.
- Çokluk (multiplicity) ifadeleri soran sorularda "tam olarak kaç" anlamını belirten doğru notasyon kullanılmalıdır.

# SQL (Structured Query Language) Kuralları

## Kapsam
Bu modül, SQL ve ilişkisel veritabanı kavramlarına ait test sorularının nasıl üretileceğini tanımlar. Sorular standart SQL (ANSI) temelinde olmalı; T-SQL (SQL Server) veya PL/pgSQL (PostgreSQL) gibi diyalekt özellikleri sorularda belirtilmelidir.

## Kod Gösterimi
1. SQL sorguları JSON içinde `"code"` alanına string olarak yerleştirilmeli, satır sonları `\n` ile ifade edilmelidir.
2. SQL anahtar kelimeleri büyük harf ile yazılmalıdır (`SELECT`, `FROM`, `WHERE`, `JOIN` vb.).
3. Tablo ve sütun adları küçük harf / snake_case ile gösterilmelidir (ör. `orders`, `customer_id`).
4. JSON string içindeki tırnak işaretleri `\"` olarak kaçırılmalıdır.

## Soru Konuları (Öncelik Sırası)

### Temel Sorgular (DML – Data Manipulation Language)
- `SELECT`, `FROM`, `WHERE`, `ORDER BY`, `LIMIT` / `TOP`
- `INSERT INTO … VALUES …`, `UPDATE … SET … WHERE …`, `DELETE FROM … WHERE …`
- `DISTINCT`, `AS` (alias), `LIKE`, `IN`, `BETWEEN`, `IS NULL` / `IS NOT NULL`
- Aggregate fonksiyonlar: `COUNT()`, `SUM()`, `AVG()`, `MIN()`, `MAX()`
- `GROUP BY` ve `HAVING` (GROUP BY sonrası filtreleme için; WHERE ile farkı)

### Join İşlemleri
- `INNER JOIN`: Her iki tabloda eşleşen kayıtlar
- `LEFT JOIN` / `RIGHT JOIN`: Eşleşmeyen kayıtların NULL olarak dahil edilmesi
- `FULL OUTER JOIN`: Her iki tablodaki tüm kayıtlar
- `CROSS JOIN`: Kartezyen çarpım
- Self-join: Bir tablonun kendisiyle birleştirilmesi
- Çoklu join: Üç veya daha fazla tablo birleştirme

### Alt Sorgular (Subquery)
- `WHERE` içinde alt sorgu (`IN`, `EXISTS`, `ANY`, `ALL`)
- `FROM` içinde türetilmiş tablo (derived table / inline view)
- Correlated (ilişkili) alt sorgu: Dış sorguya başvuran iç sorgu

### Pencere Fonksiyonları (Window Functions)
- `ROW_NUMBER()`, `RANK()`, `DENSE_RANK()` farkları
- `OVER (PARTITION BY … ORDER BY …)` sözdizimi
- `LAG()` / `LEAD()`: Önceki / sonraki satır değerine erişim
- `SUM() OVER (…)`: Kümülatif toplam

### Tablo Tanımlama (DDL – Data Definition Language)
- `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`
- `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, `NOT NULL`, `DEFAULT`, `CHECK` kısıtlamaları
- Index oluşturma: `CREATE INDEX`, `CREATE UNIQUE INDEX`

### Veri Bütünlüğü ve Normalizasyon
- 1NF, 2NF, 3NF, BCNF tanımları ve ihlal örnekleri
- Referential integrity (Referans bütünlüğü): `ON DELETE CASCADE` / `SET NULL` / `RESTRICT`
- Transaksiyonlar: `BEGIN`, `COMMIT`, `ROLLBACK`, `SAVEPOINT`; ACID özellikleri (Atomicity, Consistency, Isolation, Durability)

### İleri Kavramlar
- View (`CREATE VIEW`) ve Materialized View farkı
- Stored Procedure vs Function farkı
- Trigger: `BEFORE` / `AFTER`, `INSERT` / `UPDATE` / `DELETE` olayları
- CTE (Common Table Expression): `WITH cte_name AS (…) SELECT …`
- `UNION` vs `UNION ALL` farkı
- `CASE WHEN … THEN … ELSE … END` ifadesi

## Kod Örneği JSON Formatı
```json
{
  "id": 1,
  "text": "Aşağıdaki SQL sorgusu hangi sonucu döndürür?",
  "code": "SELECT department, COUNT(*) AS emp_count\nFROM employees\nWHERE salary > 5000\nGROUP BY department\nHAVING COUNT(*) > 3\nORDER BY emp_count DESC;",
  "choices": [
    { "label": "A", "text": "Tüm departmanların çalışan sayısını listeler.", "correct": false },
    { "label": "B", "text": "Maaşı 5000'den fazla olan ve 3'ten fazla çalışanı bulunan departmanları çalışan sayısına göre azalan sırada listeler.", "correct": true },
    { "label": "C", "text": "Maaşı 5000'den fazla olan tüm çalışanları listeler.", "correct": false },
    { "label": "D", "text": "Her departmanın toplam maaşını listeler.", "correct": false },
    { "label": "E", "text": "Sözdizimi hatası verir; HAVING WHERE'den önce yazılmalıdır.", "correct": false }
  ],
  "explanation": "WHERE satır bazlı filtreleme yaparken (GROUP BY öncesi), HAVING grup bazlı filtreleme yapar (GROUP BY sonrası). Sorgu: maaşı > 5000 olan çalışanları gruplar, 3'ten fazla çalışanı olan departmanları seçer ve azalan sırada listeler."
}
```

## Önemli Kurallar
- `WHERE` ile `HAVING` farkını test eden sorular etkili tuzak sorulardır.
- `LEFT JOIN` ve `INNER JOIN` sonuçlarının farklılaştığı NULL içeren senaryolar kullanılabilir.
- `RANK()` ile `DENSE_RANK()` arasındaki sıralama farkı (boşluk bırakma) karıştırıcı sorular üretmek için idealdir.
- Normalizasyon sorularında somut bir tablo örneği üzerinden ihlal gösterilmelidir.
- Transaksyon sorularında ACID özelliklerinden hangisinin ihlal edildiği veya hangi komutun ne işe yaradığı sorgulanmalıdır.

# Türkçe Paragraf ve Dil Bilgisi Kuralları (ÖSYM Tarzı)

1. Paragraf ve Dil Bilgisi Soruları: Türkçe dersi altındaki paragraf, cümlede anlam, sözcükte anlam ve dil bilgisi soruları için bu kurallar geçerlidir.
2. ÖSYM Formatı: Sorular YKS (TYT/AYT), KPSS, ALES ve DGS sınav formatlarında hazırlanmalıdır.
3. Uzun Paragraflar: Soru metinlerindeki öncül paragraflar yeterince uzun ve dolgun olmalıdır. Metinler akademik, felsefi, tarihi, bilimsel veya edebi bir makaleden alınmış gibi gerçekçi, okumayı ve analizi gerektiren düzeyde olmalıdır.
4. Uzun ve Çeldirici Şıklar: Seçenekler (A, B, C, D, E) tek kelimelik değil, paragrafın analizini yansıtan tam cümleler veya uzun yan cümleler halinde olmalıdır. Güçlü çeldiriciler kullanılmalıdır.
5. Soru Kökleri: Soru kökleri ÖSYM tarzı olmalıdır.
   Örnekler:
   - "Bu parçadan aşağıdakilerin hangisine ulaşılabilir?"
   - "Bu parçada asıl anlatılmak istenen aşağıdakilerden hangisidir?"
   - "Bu parçanın yazarının aşağıdakilerden hangisini söylemesi beklenemez?"
   - "Bu parçadaki altı çizili sözle anlatılmak istenen aşağıdakilerden hangisidir?"
6. Dil Bilgisi İstisnaları: Dil bilgisi sorularında paragraf veya uzun cümleler verilip, seçeneklerde "A) Girişik birleşik cümle kullanılmıştır.", "B) Yüklem geçişli bir fiildir." gibi analizler istenir.
7. Metin Biçimlendirme (Paragraf ve Soru Gösterimi):
   - Alıntılar: Metin içerisindeki alıntılar mutlaka çift tırnak ("...") içinde yazılmalıdır.
   - Paragraf Girintisi: Soru öncülündeki paragrafların başında mutlaka girinti (boşluk veya `\t` karakteri) bulunmalıdır. Eğer metin birden fazla paragraftan oluşuyorsa her paragrafın başında girinti olmalıdır.
   - Ayraç: Paragraf / öncül metin ile asıl soru kökü ("Bu parçada anlatılmak istenen...", vb.) arasına görsel bir ayraç (örneğin `\n\n---\n\n` veya sadece `\n\n`) konarak metin ile soru kökü birbirinden net bir şekilde ayrılmalıdır.
   - Önemli İfadeler: Metin içindeki altı çizili veya vurgulanmak istenen söz öbekleri tek tırnak ('...') veya kalın (bold) ifadelerle belirginleştirilmelidir.
